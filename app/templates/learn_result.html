Step 2 — Callback Received

{{ debug.provider | capitalize }} redirected the user back with these query parameters:

code {{ debug.callback_code }}
state {{ debug.callback_state }} verified
What just happened?

{{ debug.provider | capitalize }} redirected the user back to your redirect_uri with two query parameters: code and state.

The code is an authorization code. It is short-lived (usually 10 minutes) and can only be used once. It is NOT an access token.

The state matches what your app sent in Step 1. If it did not match, your app would reject the request to prevent CSRF attacks.

Step 3 — Token Exchange

Your server exchanged the code for an access token. This happens server-to-server — the user's browser is not involved.

{% for key, value in debug.token_request_body.items() %}{{ key }}: {{ value }} {% endfor %}

{% for key, value in debug.token_response_raw.items() %}{{ key }}: {{ value }} {% endfor %}
What just happened?

Your app made a server-to-server POST request to the token endpoint. This request includes the client_secret, which proves your app's identity. The secret should never be exposed to the frontend.

The code from Step 2 was exchanged for an access_token. The code can only be used once — if someone intercepted it, they cannot reuse it.

The access_token is what your app uses to make API calls on behalf of the user.

What just happened?

Every provider returns user data differently. GitHub uses login and avatar_url. Google uses name and picture.

The normalize_userinfo() method maps provider-specific fields into a standard shape: id, name, email, avatar. This abstraction lets your app work with any provider without changing the rest of your code.
